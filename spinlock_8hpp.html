Detailed Description

Spinlock implementation using std::atomic_flag.

In most situations, application should just use std::mutex. This lets OS scheduler to decide when waiting for lock has been too long and suspend waiting core. See http://stackoverflow.com/questions/5869825/when-should-one-use-a-spinlock-instead-of-mutex for more information.

If for specific situation spinlock still makes sense, this implementation allows to customize loop while waiting by using callable yielding policy as parameter for spinlock_t::lock(SpinYield). There are some pre-implemented policies (busy_spin, yield_spin, adaptive_spin).

Note
Before deciding whether to use spinlock_t or std::mutex do profiling.

void sal::adaptive_spin ( size_t  iter_count)
inline

Adaptive remaining timeslice yielding policy for spinlock_t::lock(SpinYield).

Depending on how many times spinlock_t::lock() has spinned, it yields differently:

  • iter_count <= BusySpinCount Busy spinning
  • iter_count > BusySpinCount Yield remaining timeslice
  • iter_count > 2x BusySpinCount Sleep i.e. let OS scheduler suspend core if it decides so
